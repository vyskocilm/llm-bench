# Deep-thinking Bench (Track A) — Design

**Author**: claude-opus-4-7
**Date**: 2026-05-15
**Status**: design approved; pending implementation plan

## 1. Motivation

The existing local OSS LLM benchmark (`bench/lm-eval-run.sh`, results in `bench/REPORT.md`) measures models with `chat_template_kwargs.enable_thinking=false` — i.e. "quick-chat" mode. This is the right default for cross-family fairness and interactive iGPU use, but it leaves the deep-reasoning capability of Gemma 4 and Qwen 3 / 3.6 unmeasured.

Public Qwen 3 reports show `enable_thinking=true` lifts scores by **+15-20 pp on gsm8k/humaneval/mmlu_pro and +30-45 pp on MATH**. Our own n=164 relaxed-humaneval runs (2026-05-14) confirm that the thinking-off ceiling on humaneval for Qwen 3 models is genuinely ~0.32 — there is no "harness artifact" hiding higher scores. The deep-thinking lift is therefore the largest single uncaptured signal in the bench suite.

This design specifies a self-contained "deep-bench" track that runs the same models with thinking-on, under a budget that fits two overnight cycles.

## 2. Goals

- Measure the thinking-on score lift on two thinking-capable models that bracket the available size range (smallest + biggest among the local fleet).
- Produce defensible numbers (n ≥ 100, stderr ≤ ±5 pp at p=0.7) for headline tasks.
- Keep the new track strictly separate from the existing fast-chat report. Two reports, one repo.
- Fit each sub-profile inside a single overnight run (≤ 24h wall-clock).

## 3. Non-goals

- Cross-family ranking among thinking-capable models. (Two models is not enough; that needs a follow-up sweep with the other thinking-capable models.)
- A unified score that combines thinking-off and thinking-on results. The two are reported side-by-side, not merged.
- Track B (qualitative / vibes gallery). Captured as future work only — see §11.
- Re-running the entire 13-model fleet at thinking-on. Out of budget and out of point: non-thinking models have no thinking mode.
- Modifying `bench/lm-eval-run.sh` or `bench/REPORT.md`. The fast-chat track is frozen for this work.

## 4. Scope

### 4.1 Models (α: size-bracket pair)

| model | total / active params | quant | GGUF size |
|-------|------------------------|-------|-----------|
| `gemma-4-e2b` | 5B / 5B (dense small) | Q4_K_M | 2.9 GiB |
| `qwen3.6-35b-a3b` | 35B / 3B (MoE) | Q5_K_M | ~24 GiB |

Rationale: smallest vs biggest thinking-capable in the local fleet. Both are iGPU-feasible (e2b fits trivially; qwen3.6-35b is MoE so the 3B active path is memory-bandwidth-friendly).

### 4.2 Sub-profiles

| sub-profile | tasks | n per task | est. wall-clock (both models, sequential) |
|-------------|-------|-----------|------------------------------------------|
| `deep-reasoning` | gsm8k, humaneval_instruct_relaxed, mmlu_pro | 200 / 164 / 200 | ~11h |
| `deep-math` | hendrycks_math (random 100 of 5000 test) | 100 | ~10h |

`humaneval_instruct_relaxed`'s n is the natural maximum (the task has 164 prompts total). `math` n is set to 100 because the thinking-on lift on math is +30-45 pp — well above the noise floor at n=100 — and a higher n would push past the 24h overnight budget on qwen3.6-35b-a3b.

### 4.3 Runtime config (per request)

- `enable_thinking=true` (via `chat_template_kwargs` in `extra_body`)
- `max_gen_toks=8192` (sufficient for ~95% of thinking traces; the 5% tail truncates and counts as fail)
- `temperature=0.6` (Qwen-recommended for thinking mode; not 0.0, since deterministic decoding causes thinking-trace loops in some prompts)
- `do_sample=true`
- `until=[]` (empty stop list — strict stops fire mid-`<think>`)
- `seed=42` (run reproducibility)
- llama-server `--parallel 4` (concurrency inside the server; same as fast-chat)

```

│   ├── tasks/              # NEW task YAML overrides
│   │   ├── gsm8k_deep.yaml
│   │   ├── humaneval_deep.yaml
│   │   ├── mmlu_pro_deep.yaml
│   │   └── math_deep.yaml
│   ├── deep-reasoning/             # results
│   │   ├── gemma-4-e2b/<task>/{driver.log, lm-eval.log, server.log, results.json}
│   │   └── qwen3.6-35b-a3b/<task>/...
│   ├── deep-math/                  # results
│   │   ├── gemma-4-e2b/<task>/...
│   │   └── qwen3.6-35b-a3b/<task>/...

└── future/

```

Key separation guarantees:

- DEEP-REPORT.md is its own document. REPORT.md gets a one-line link only.
- Track A results never get mixed into the fast-chat tables.
- The existing `lm-eval-run.sh` is untouched. `deep-bench.sh` reuses no code from it (intentionally — keeps the existing script simple).

## 6. Components

### 6.1 Task YAMLs (`bench/deep/tasks/*.yaml`)

Each inherits from the upstream task and overrides only the deep-bench-specific knobs.

| YAML | inherits from | overrides |
|------|---------------|-----------|
| `gsm8k_deep.yaml` | `gsm8k` | `generation_kwargs: {until: [], max_gen_toks: 8192, temperature: 0.6, do_sample: true}` |
| `humaneval_deep.yaml` | `humaneval_instruct_relaxed` | same as above |
| `mmlu_pro_deep.yaml` | `mmlu_pro` | `generation_kwargs: {max_gen_toks: 8192, temperature: 0.6, do_sample: true}` (mmlu_pro has no stop list to override) |
| `math_deep.yaml` | `hendrycks_math` (full MATH test set, 5000 problems; cap to 100 via `--limit`) | `generation_kwargs: {until: [], max_gen_toks: 8192, temperature: 0.6, do_sample: true}, num_fewshot: 0` |

Each YAML is committed in `bench/deep/tasks/` (source of truth) and copied at driver runtime into `~/.local/share/uv/tools/lm-eval/lib/python3.14/site-packages/lm_eval/tasks/<area>/` so that lm-eval's `include:` directive resolves relative to the parent task's directory. This mirrors the workaround used for `humaneval_instruct_relaxed`.

### 6.2 EXTRA_BODY config

Hard-coded inside `deep-bench.sh`, applied identically to both models:

```bash
DEEP_EXTRA_BODY='{"chat_template_kwargs":{"enable_thinking":true}}'
```

The driver exports `EXTRA_BODY` as an env var before each lm-eval call. The existing `LM_EVAL_EXTRA_BODY` plumbing in `openai_completions.py` picks it up. The driver does **not** rely on `lm-eval-run.sh`'s `MODEL_EXTRA_BODY` lookup table.

### 6.3 Harness patch — robust `<think>` post-strip

The existing patch in `~/.local/share/uv/tools/lm-eval/lib/python3.14/site-packages/lm_eval/models/openai_completions.py::parse_generations` strips a single leading `<think>...</think>` block. This is insufficient for thinking-on output because models emit:

- Multiple `<think>` blocks (mid-response thinking).
- Unclosed `<think>` when truncated at `max_gen_toks` (block runs to EOS with no `</think>`).
- Channel-tag variants `<|channel|>thought<|channel|>` (Gemma 4 MoE).

Replace the existing leading-only strip with a 4-pattern strip that is a strict capability superset:

```python
import re
# Closed thinking blocks — strip all occurrences, non-greedy
text = re.sub(r"<think>.*?</think>", "", text, flags=re.DOTALL)
text = re.sub(r"<\|channel\|>.*?<\|channel\|>", "", text, flags=re.DOTALL)
# Unclosed <think> (truncated at max_gen_toks) → drop from open tag to EOS
text = re.sub(r"<think>.*", "", text, flags=re.DOTALL)
text = re.sub(r"<\|channel\|>.*", "", text, flags=re.DOTALL)
```

Harmless on fast-chat output (no thinking tags present → no-op). Unit-tested via `bench/deep/test_strip.py`.

### 6.4 Driver — `bench/deep/deep-bench.sh`

Public surface:

```bash
./deep-bench.sh deep-reasoning   # runs gsm8k + humaneval + mmlu_pro, both models
./deep-bench.sh deep-math        # runs math, both models
./deep-bench.sh smoke            # n=5 sanity check on smollm3-3b
```

Internal structure:

- Model registry (alias → gguf path, tokenizer ref, ctx-size).
- Task list per sub-profile.
- For each model: start `llama-server` on :18080, poll `/v1/models`, run each task sequentially via `uv run lm-eval`, kill server.
- Skip-existing: if `bench/deep/<sub-profile>/<model>/<task>/results.json` exists and is valid JSON, skip that task.
- Per-task EXTRA_BODY env injection.
- Server-side `--parallel 4` for concurrency.

Outputs per task: `{driver.log, lm-eval.log, server.log, results.json}` under `bench/deep/<sub-profile>/<model>/<task>/`.

### 6.5 Aggregator — `bench/deep/aggregate-deep.py`

Reads all `results.json` under `bench/deep/`, joins against fast-chat scores in `bench/lm-eval/<model>/results.json`, regenerates `bench/deep/DEEP-REPORT.md` in place. Markdown is never hand-edited; JSON is the source of truth.

DEEP-REPORT.md layout: one table per task with columns `model | fast-chat score | deep-thinking score | Δ pp | stderr`.

## 7. Data flow

```
deep-bench.sh deep-reasoning
        │
        ▼
for each model in [gemma-4-e2b, qwen3.6-35b-a3b]:
    1. start llama-server with --jinja --no-warmup --parallel 4
    2. poll /v1/models until ready
    3. for each task in [gsm8k_deep, humaneval_deep, mmlu_pro_deep]:
         export EXTRA_BODY='{"chat_template_kwargs":{"enable_thinking":true}}'
         uv run lm-eval ... --tasks <task> --limit <n>
         → writes results.json to bench/deep/deep-reasoning/<model>/<task>/
    4. kill llama-server
        │
        ▼
post-pass: aggregate-deep.py → DEEP-REPORT.md
```

Guarantees:

- One model in GPU at a time; server stops + restarts between models.
- Server stays up across tasks within a sub-profile (avoids ~3 min reload per task).
- Tasks within a sub-profile run sequentially; concurrency is from `--parallel 4` inside the server.
- DEEP-REPORT.md pulls fast-chat scores read-only from `bench/lm-eval/<model>/results.json`; does not modify them.

## 8. Error handling

| risk | mitigation |
|------|-----------|
| Unclosed `<think>` at `max_gen_toks` | post-strip drops `<think>` to EOS; result counts as fail. No fake-rescue. |
| HTTP timeout on long thinking | bump aiohttp `timeout=3600` in `openai_completions.py` for deep-bench calls. |
| Vulkan device lost (oversized KV cache) | per-model `MODEL_CTX` table required; both deep-bench models have native 131072 ctx, run at `ctx-size=131072`. |
| EXTRA_BODY not applied | driver hard-codes the env per model; doesn't rely on alias-keyed lookup. Misapplication fails fast with HTTP 400 at first request. |
| Thinking-loop on MATH ("I cannot solve this") | `max_gen_toks=8192` is the hard cap; fail counts as fail. |
| mmlu_pro letter-parse fail (CoT inside `<think>`) | mmlu_pro's parser handles "answer is A" / "A)" / "final answer: A". If smoke shows issue, add small fallback parser that pulls last `[A-J]` token. Implement only if needed. |
| User kills mid-run | each task's results.json is written atomically by lm-eval. Skip-existing logic in driver resumes from last completed task. |
| Server crash mid-run | driver detects llama-server exit-code != 0, dumps last 200 lines of server.log to driver.log, restarts with fresh server for next task. |
| Power loss / OOM | no special handling; re-run from CLI, idempotent skip-existing logic handles partial state. |

## 9. Testing / validation

Gate the overnight run on cheap smoke tests first.

| stage | what | pass criteria | failure action |
|-------|------|---------------|----------------|
| **0** | tokenizer pre-flight via `AutoTokenizer.from_pretrained` for both models | loads in < 5s | install `sentencepiece` / fix repo ref |
| **1** | `curl :18080/v1/chat/completions` with `enable_thinking=true`, simple math prompt | response contains `<think>` block + correct numeric answer | check llama-server jinja template support; bump max_tokens |
| **2** | `./deep-bench.sh smoke` — all 4 tasks at n=2 on smollm3-3b | finishes < 10 min, emits scores table | catch task-yaml syntax, EXTRA_BODY plumbing, strip regex |
| **3** | smollm3-3b smoke scores sanity check | gsm8k > 0.5, humaneval > 0.2 | if zero, strip-regex or stop list is broken |
| **4** | gemma-4-e2b on gsm8k at n=10, thinking-on | < 10 min wall-clock; score > thinking-off baseline (~0.85) | if score ≤ baseline, EXTRA_BODY not being applied |
| **5** | qwen3.6-35b-a3b on gsm8k at n=10 | < 30 min wall-clock; score check | server stability under thinking-on for 35B model |
| **6** | full `./deep-bench.sh deep-reasoning` then `deep-math` | as planned | already-validated path |

Stages 0-5 sum to ~1h. Cheap insurance vs blowing a 21h overnight on a config bug.

`bench/deep/test_strip.py`: 6-8 hand-crafted strings (closed, unclosed, nested, channel-tag, no-thinking) through the strip function, asserts expected output. Runs < 1s.

## 10. Success criteria

- [ ] `deep-bench.sh smoke` passes on smollm3-3b in < 10 min.
- [ ] `bench/deep/test_strip.py` passes.
- [ ] `deep-reasoning` sub-profile completes for both models in < 14h wall-clock.
- [ ] `deep-math` sub-profile completes for both models in < 12h wall-clock.
- [ ] DEEP-REPORT.md shows positive thinking-on Δ on all 4 reasoning tasks for at least one model.
- [ ] DEEP-REPORT.md is reproducible from `results.json` files via `aggregate-deep.py`.
- [ ] REPORT.md adds exactly one line pointing to DEEP-REPORT.md; no other changes.
