# llama.cpp on Radeon 860M (gfx1151 / Krackan Point)

Optimized localhost llama-server build for Ryzen AI 7 PRO 350 + Radeon 860M
iGPU. Backend: Vulkan via RADV. ROCm is not used: upstream rocBLAS images ship
without Tensile kernels for gfx1150/1151, so init segfaults or the device is
filtered out. Vulkan bypasses the rocBLAS path entirely.

Quick links:

- **Results**: [`bench/REPORT.md`](bench/REPORT.md) (fast-chat / `enable_thinking=false` mode, all bench'd models)
- **Deep-thinking results**: [`bench/deep/DEEP-REPORT.md`](bench/deep/DEEP-REPORT.md) (Gemma 4 + Qwen 3.6 with thinking-on; regenerated by `bench/deep/aggregate-deep.py`)
- **Models manifest**: [`MODELS.md`](MODELS.md) (HF source + SHA256 per GGUF; weights are NOT in this repo)
- **Licence**: [`LICENSE`](LICENSE) (Apache-2.0) — applies to scripts/configs/docs in this repo only. Each model weight inherits its upstream repo's licence (Apache-2.0 / MIT for all bench'd models).

## Build

Inside the distrobox container (`distrobox enter <name>`):

```sh
./build.sh
```

Lists missing zypper packages on first failure. Install them on the host
(immutable Tumbleweed — `sudo zypper in ...`), re-enter the distrobox, repeat
until clean. Output: `./dist/bin/{llama-server,llama-cli,llama-bench}`.

## Image

On the host (or distrobox — both share rootless podman storage if configured
that way):

```sh
podman build -t llama-vulkan:latest .
```

`Dockerfile` does no compilation — it only installs the RADV runtime
(`libvulkan_radeon`, `libvulkan1`, `libgomp1`, `libcurl4`, `libssl60`,
`libcrypto57`, `ca-certificates`) and copies `./dist/bin/*`. Note: current
Tumbleweed libcurl is linked against LibreSSL 4.x (SONAMEs `libssl.so.60` /
`libcrypto.so.57`), *not* OpenSSL 3 — don't substitute `libopenssl3`.

## Model

```sh
./pull-model.sh
```

Drops `models/qwen2.5-3b-instruct-q4_k_m.gguf` (~1.9 GiB). Verifies sha256
against the HF LFS `X-Linked-Etag` header. Override with `MODEL=...` and
`FILE=...` env vars for other GGUFs.

## Run (quadlet)

```sh
mkdir -p ~/.config/containers/systemd
cp llama.container ~/.config/containers/systemd/
systemctl --user daemon-reload
systemctl --user start  llama.service
systemctl --user enable llama.service    # auto-start on login
loginctl enable-linger "$USER"           # auto-start without a login session
journalctl --user -u llama.service -f
```

Look for `llama_prepare_model_devices: using device Vulkan0 (AMD Radeon 860M
Graphics (RADV KRACKAN1))` during startup; that confirms the iGPU is in use.
If no `Vulkan0` line appears (model loads but no device-selection log),
Vulkan enumeration failed — check render-group propagation (`ls -l /dev/dri`
inside the container; user must have access to `renderD*`).


Or the OpenAI-compatible endpoint at `http://127.0.0.1:8080/v1` (chat
completions).

## Clients

`curl` is just the smoke-test path. Three more useful ways to talk to the
server:

### Built-in web UI

llama-server ships a chat UI at `/`. No config needed:

```sh
xdg-open http://127.0.0.1:8080/
```

Streams, supports system prompts and sampling knobs.

### crush (OpenAI-compatible TUI)

Point crush at the `/v1` endpoint. In `~/.config/crush/crush.json`:

```json
{
  "providers": {
    "local-llama": {
      "type": "openai",
      "base_url": "http://127.0.0.1:8080/v1",
      "api_key": "dummy",
      "models": [
        { "id": "qwen2.5-3b", "name": "Qwen2.5-3B (local, gfx1151)" }
      ]
    }
  }
}
```

The model id `qwen2.5-3b` matches the quadlet's `--alias qwen2.5-3b`. Pick
`local-llama/qwen2.5-3b` in crush's model picker.

### claude (Anthropic CLI / Claude Code)

`claude` speaks the Anthropic Messages API, not OpenAI Chat Completions —
it won't talk to llama-server directly. Workarounds:

- Run an adapter like `claude-code-router`, `anthropic-proxy`, or
  `litellm --model openai/qwen2.5-3b` in proxy mode, then
  `ANTHROPIC_BASE_URL=http://127.0.0.1:<proxy-port> claude`.
- For serious work, keep `claude` pointed at the real Anthropic API —
  Qwen2.5-3B is too small to be a useful drop-in.

### Quick OpenAI-compat probe

Confirm the path the clients use:

```sh
curl -fsS http://127.0.0.1:8080/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"qwen2.5-3b","messages":[{"role":"user","content":"two sentences on RADV"}]}' \
  | jq -r '.choices[0].message.content'
```

If that returns text, crush will work.

## Benchmark

Two layers:

- **`dist/bin/llama-bench`** — upstream micro-benchmark (tokens/s, prompt vs.
  generation). Quick GPU/CPU compare against a running server:

  ```sh
  podman exec -it llama-server \
    llama-bench -m /models/qwen2.5-3b-instruct-q4_k_m.gguf -ngl 999 -t 8
  ```

  (Run inside the same container so it uses the same Vulkan stack.)

- **`bench/lm-eval-run.sh`** — lm-evaluation-harness sweep against the
  running llama-server. Strict-OSI gate (Apache-2.0 / MIT only). Three
  profiles via `BENCH_PROFILE`: `quick` (n=30, ~30 min), `core` (n=250,
  ~2-3 h per model), `full` (n=full, hours-days). Results land in
  `bench/lm-eval/<alias>/` as JSON; rolled-up into [`bench/REPORT.md`](bench/REPORT.md).

- **`bench/deep/deep-bench.sh`** — deep-thinking track (Track A in the
  spec). Runs Gemma 4 / Qwen 3.6 with `enable_thinking=true` against
  gsm8k / humaneval / mmlu_pro / hendrycks_math500 in their own
  sub-profiles. Self-contained; rolled-up into
  [`bench/deep/DEEP-REPORT.md`](bench/deep/DEEP-REPORT.md). See spec at
  `docs/superpowers/specs/2026-05-15-deep-bench-design.md`.

## Troubleshooting

- **`Failed to open /dev/dri/renderD128: Permission denied`** — the render
  gid did not propagate. The quadlet uses `--userns=host` so the container
  inherits all host supplementary groups; verify `id` inside the container
  lists the host's `render` gid. If not, the host user is not in `render` —
  `sudo usermod -aG render,video "$USER"` on the host and re-login.
- **`vulkan: no devices found`** — RADV ICD not in the image. Check
  `libvulkan_radeon` is installed in the runtime layer.
- **CPU-only inference** — `-ngl 999` not passed, or model offload disabled.
  Inspect `journalctl --user -u llama.service` for the device-selection log
  line.
- **`HSA_OVERRIDE` / rocBLAS errors** — wrong image; this stack is
  intentionally Vulkan-only. Do not enable ROCm.
